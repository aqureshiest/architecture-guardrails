Opinionated, battle-tested guardrails for building high-quality systems — grounded in real industry lessons, not surface-level advice. From code craftsmanship through AI-augmented workflows.

Section 01

Code Craftsmanship & Developer Pride

Code is written once and read hundreds of times. Optimise relentlessly for the reader, not the writer.

Core Insight

Ward Cunningham described technical debt as the difference between the understanding you have now and the understanding embedded in the code. It accumulates every time the code fails to reflect current domain knowledge — and like financial debt, it compounds. The interest is paid in every sprint: slower delivery, higher defect rate, harder onboarding.

📐 Design for the Consumer

Before writing a function or module, ask: how should a caller use this? A clean interface makes the right thing easy and the wrong thing hard. Every public method is a contract — design it from the outside in, not the inside out.

  • Write the call site first — if usage feels awkward, the design is wrong
  • Expose the minimum necessary; default everything to private. Shrinking public surface is often more valuable than adding features
  • A function that does one thing at one level of abstraction is always easier to use, test, and replace

🪟 The Broken Window Effect

One hack, one dead code block, one misleading name signals that nobody cares. Teams lower their standards to match the existing bar, and discipline collapses quickly once the first exception is made.

  • Boy Scout Rule: leave code better than you found it — even one renamed variable counts
  • Tech debt discovered during feature work must be filed, tracked, and paid down
  • Never commit code you'd be embarrassed to defend in a post-mortem

✅ Code Review Culture

Reviews are the highest-leverage quality practice — but only when genuinely about correctness and clarity, not gatekeeping. The reviewer learns as much as the author.

  • Ask: will a stranger understand this in six months? Are errors handled or silently swallowed?
  • Style wars belong in linter config, not review comments
  • Rotate reviewers deliberately to avoid knowledge silos
  • When the same issue surfaces repeatedly in reviews, convert it to a guardrail

🎯 Names Reveal Intent

If you need a comment to explain a variable, the variable is misnamed. Names should make code read as a sentence about the problem domain.

  • Use the domain's vocabulary — align with product and business language
  • Command-Query Separation: a function either changes state or returns data, never both
  • Rename aggressively when understanding evolves — stale names are debt

Section 02

Simplicity & Avoiding Over-Engineering

Complexity is the root cause of most software failures. It is never free, always compounds, and is much easier to add than to remove.

Core Insight

Rich Hickey's distinction: simple means one responsibility, low cognitive load. Easy means familiar. We routinely choose easy over simple — ORMs, frameworks, clever abstractions — because it feels productive. Easy-now compounds into hard-later. The real cost of a complex system isn't building it; it's every future engineer who must hold it in their head.

💸 Complexity Is Debt With Interest

Every layer, pattern, or service added speculatively is a bet on a future that often doesn't arrive. The carrying cost is paid in every sprint — slower onboarding, harder debugging, more things to break.

  • Accidental complexity is always avoidable; essential complexity should be isolated, not spread
  • Every new abstraction needs a business case: what problem does this solve right now?
  • Measure complexity by onboarding time — a new engineer productive in a day is the target

🏗 Start Simple, Split Deliberately

The distributed monolith is the most damaging pattern: all the operational cost of distributed systems with none of the independence. Start with a modular monolith; extract components only when you have a specific, demonstrated need.

  • Valid reasons to split: independent scaling, independent deployability, compliance boundary
  • Invalid reasons: "the code looks big," "that's what modern systems do"
  • Conway's Law is real — your architecture mirrors your team structure whether you plan for it or not

🚫 Premature Abstraction & YAGNI

DRY is about knowledge duplication, not text duplication. Two functions that look similar but represent different concepts should not be merged — they will diverge, and you'll pay for the artificial coupling.

  • Rule of Three: wait for the third instance before abstracting
  • Sandi Metz: duplication is far cheaper than the wrong abstraction
  • YAGNI applies to infrastructure too — don't provision for 10× load you don't have

📉 Simplicity Health Check

A few quick signals on whether your system has stayed grounded or started accumulating complexity for its own sake.

TDD & Shift-Left Quality

Testing is not a phase — it is a design practice. Untestable code is always poorly designed code.

Core Insight

Kent Beck's real insight was not "write tests before code" — it was: if you write a test first, you are forced to use your own API before it exists, and bad APIs reveal themselves immediately. TDD is a design feedback loop. NIST data shows defects cost 1× to fix in development and 100× in production. Shift-left is basic economics, not a philosophy.

🔺 Testing Pyramid

Each layer plays a different role. Get the ratios right and you have a fast, trustworthy suite. Get them wrong and you have either slow CI or false confidence.

  • Unit: test behavior, not implementation; Arrange-Act-Assert always
  • Integration: use a real database in a container — mocking the DB is a reliable way to ship bugs
  • E2E: critical user paths only; flaky tests are bugs, not inconveniences to skip
  • Contract: consumers publish expectations, providers verify them on every build

🧬 Beyond Coverage

Coverage measures which lines ran — not whether tests catch bugs. A suite with 90% coverage and 60% mutation score is telling you the coverage was gamed.

  • Mutation testing introduces small bugs and verifies tests fail — surviving mutants reveal exactly what's unverified
  • Property-based testing generates thousands of inputs, finding bugs months of manual testing miss
  • Fitness functions enforce architecture constraints in CI — dependency direction, performance budgets, security policies

⬅️ Shift Left Gates

Catch problems where they're cheapest to fix. Each gate in the pipeline narrows the surface of issues that can reach production.

  • Pre-commit: formatting (automated, never debated), static analysis, secret scanning
  • Pull request: full test suite, coverage as a hard fail, SAST, dependency CVE scan
  • Deploy: canary with automated SLO-based rollback, smoke tests before traffic routing
  • Flaky tests are bugs — fix or delete them, never retry

Section 04

Third-Party Integration Patterns

Every external dependency is a risk you didn't design, a failure mode you don't control, and a security surface you must actively protect.

Fallacies of Distributed Computing

Peter Deutsch: the network is reliable; latency is zero; bandwidth is infinite; the network is secure; topology doesn't change. Every integration is built on these false assumptions. Design for their failure from day one — not after the first production incident teaches you to.

⚡ Resilience Patterns

A single slow dependency without protection will exhaust your thread pool and cascade failures through your entire system. Resilience is not a feature — it's the minimum viable defense.

  • Circuit breaker: open on failure threshold, fail fast, half-open probe to recover. Tune per integration
  • Retry with backoff and jitter: 1s, 2s, 4s — not 1s, 1s, 1s. Only retry idempotent operations
  • Timeouts on everything: connection and read timeouts, set separately. Always shorter than your own SLA

🧱 Anti-Corruption Layer

Prevent the external system's concepts, naming, and structure from bleeding into your core business logic. When a vendor changes their API, only the ACL changes.

  • One dedicated module owns all communication with the third party — the rest of the system stays vendor-agnostic
  • Translate external types to internal domain types at the boundary; never use vendor DTOs in business logic
  • Swapping vendors should touch only the ACL — that is the test of a well-formed boundary

🔐 Secrets & Credentials

Credentials are the most valuable thing in your system. Treat them like cash, not configuration.

  • API keys never in code or committed env vars — use a secrets manager like Vault or AWS Secrets Manager
  • Least-privilege: each component has only the scopes it actually uses
  • Automated rotation, tested under load — if you can't rotate without downtime, you can't rotate under pressure
  • Audit-log every secret access; anomaly patterns catch compromises early

📬 Webhooks & API Versioning

Webhooks look simple but have subtle failure modes. Version drift across vendors is a slow leak that becomes an outage when their old version is finally removed.

  • Return 2xx immediately and process asynchronously — never block a webhook with long work
  • Idempotency is mandatory: the same event delivered twice must produce the same result
  • Verify HMAC signatures — never trust an incoming webhook without validation
  • Track which vendor API version each component consumes; upgrade proactively, not reactively

Section 05

Database Performance Patterns

The database is almost always the bottleneck and the hardest part of a system to change once live. Treat every query, schema decision, and migration as a long-term commitment.

Industry Lesson

The most common cause of production performance incidents is not hardware — it is queries that worked fine at 10k rows and catastrophically at 10M. A missing index turns O(log n) into O(n). Every new query and schema change must be reviewed as if the table already has 100× its current row count.

⚡ Query Discipline

Most performance incidents are not infrastructure problems — they're query problems that became visible only at scale. Catch them at review time.

  • Run EXPLAIN ANALYZE on every new query before merge — sequential scan on a large table is a merge blocker
  • Covering indexes include all columns the query needs so it never touches the table
  • Partial indexes target only the rows queries actually filter on
  • N+1 is a bug — never query inside a loop; use eager loading, batching, or DataLoader
  • Paginate everything variable-length with keyset (cursor) pagination, not offset

🔄 Safe Schema Migrations

Adding a non-null column to a 200M-row table can take an exclusive lock for the full backfill. This pattern has caused 45-minute outages at companies without migration discipline.

  • Expand/Contract: add nullable column, backfill in batches, switch reads, drop old column. Never skip phases
  • Use online tools for large tables — gh-ost for MySQL, pg_repack for Postgres
  • Test migrations on a production-sized dataset before running in production

🔒 Transactions & Connections

Long-running transactions and unbounded connections turn small traffic spikes into full database outages. Both are easy to prevent and almost impossible to recover from under load.

  • Keep transactions short — do all preparation outside the transaction
  • Never wait for external I/O (HTTP calls, file reads) inside a transaction
  • Alert on transactions open longer than 30 seconds — almost always a bug
  • Use a server-side connection pool like PgBouncer; size application pools deliberately
  • Route reporting and analytics to read replicas — protect the primary write path

Observability & Monitoring

Monitoring tells you whether a known condition occurred. Observability lets you answer questions you didn't know to ask when you built the system.

Core Distinction

Charity Majors: monitoring is for known unknowns — you define the condition and wait. Observability is for unknown unknowns — something fails in a way you didn't anticipate, and you can still interrogate the system's state. Most production systems are monitored but not observable. The gap is where incidents live for hours.

📝 Structured Logging

Logs are the narrative of what your system did. Structured logs make that narrative searchable and queryable instead of a wall of text.

  • JSON only — every log line is a structured object, never a concatenated string
  • Mandatory fields: timestamp (ISO 8601), service name, level, correlation/trace ID
  • Propagate correlation ID through every hop: HTTP, async messages, background jobs
  • Never log PII or credentials. Log at decision boundaries, not every line

📈 Metrics & Four Golden Signals

Google SRE's four signals are the minimum viable metric set for any production system. Everything else is supplemental.

  • Latency: p50, p95, p99 separately — averages always lie
  • Traffic: requests/sec, messages/sec — the demand baseline
  • Errors: rate of failures; distinguish 4xx (client) from 5xx (server)
  • Saturation: CPU, memory, connection pool, queue depth
  • Pair technical metrics with business metrics; revenue/min correlates with system health

🔍 Distributed Tracing

A trace tells the complete story of a request as it travels through your system. Without it, debugging a slow request is archaeology; with it, navigation.

  • OpenTelemetry as the standard — vendor-neutral, portable across backends
  • Auto-instrument HTTP clients, DB drivers, queue consumers
  • 100% sampling for errors; 1–10% for healthy traffic
  • Trace IDs in error responses — support hands off the full debug path instantly
  • Add business context to spans: user ID, order ID, feature flag state

🔔 Alerting & Post-Mortems

Alert fatigue is an engineering failure. Every alert that fires without action trains engineers to ignore alerts — including the one that matters.

  • Alert on symptoms, not causes: "error rate elevated," not "CPU at 70%"
  • Every alert has a runbook link — no undocumented alert ever merges
  • An alert that fires twice a week without an incident is noise; fix or delete it
  • Post-mortems: blameless, 5 Whys to systemic root cause, time-bound action items, shared across teams
SLO / SLI / SLA

SLI is the measurement. SLO is the target — "99.5% of requests under 200ms." SLA is the contract with consequences. Set SLOs before launch, not after the first incident. Error budgets are what's left over: if the budget is exhausted, stop new features and fix reliability. Don't set 99.99% because it sounds impressive — set the minimum acceptable level.

Section 07

AI-Augmented & Agentic Development

AI agents don't introduce new failure modes. They amplify the ones already in the system — and they amplify quietly. The slow CI you tolerated becomes a deadlock. The review you skimmed becomes a vulnerability. The ownership gap you papered over becomes a post-mortem.

The Constraint Isn't the Model. It's the Org.

The real question isn't whether AI writes good code. It's whether your system can absorb code at this volume without losing the plot. Engineers shift from primary authors to system designers, context providers, and judgment appliers. The leverage moves from typing speed to clarity of thinking — but only if the SDLC around them is ready to hold the weight. Build the system that earns the leverage. Don't hand out keys ahead of it.

🏗 The System Has to Earn the Leverage

Every latent weakness in the SDLC — flaky tests, slow pipelines, ambiguous ownership — stops being a papercut and becomes load-bearing the day agents start landing PRs at machine speed. Ramp access with the system's capacity, not with engineer demand.

  • CI you trust under 10 minutes — anything slower collapses into review-by-thumbs
  • Tests that actually catch regressions — agents pass weak tests by writing weak code
  • CODEOWNERS that names someone who'll pick up the phone — fuzzy ownership stops being benign
  • Observability that catches what review misses — silent regressions compound at PR throughput
  • Agent context files treated as code, not lore — stale context produces stale output, confidently

👁 Plausible Is Not Correct

AI output reads with the cadence of a senior engineer regardless of whether it's right. The clean prose, the confident naming, the well-formatted diff — all of it removes the visual friction that normally triggers careful reading. The instinct to skim grows. The cost of skimming grows faster.

  • Review the diff, not the prompt — what shipped is what matters
  • If you can't explain why a change is needed, you can't approve it
  • Watch for invented APIs, hallucinated conventions, plausible-but-wrong logic, unnecessary abstraction
  • Push back on PR size — long PRs get harder to review at the same rate they get easier to generate
  • Mutation testing on AI-touched code — agents will pass tests by doing the wrong thing, and you'll never know

🎯 Different Code, Different Leashes

A bad refactor wastes an afternoon. A bad migration takes down production. Treating all AI work the same flattens that distinction, and the flattening always tilts toward more risk, not less. Encode the rules; don't rely on memory.

  • Reasonable to delegate: tests, in-file refactors, doc updates, mechanical changes with clear acceptance criteria
  • Requires human design: migrations, infra, secrets, auth, public APIs, anything crossing service boundaries
  • Use CODEOWNERS and branch protection to enforce — convention does not survive contact with velocity
  • Agents work on branches, never on main. Auto-merge from AI is a red line, not a configuration option

🔒 The AI Didn't Write It. You Did.

The model didn't get paged. You did. The model isn't in the post-mortem. You are. Accountability doesn't transfer to the tool, no matter how capable the tool gets — and the org's job is to make that accountability honest by gating access on readiness, not on requests.

  • "The AI wrote it" is not an explanation. Ever
  • Access ramps with the system's capacity to absorb it — the gate is operational, not political
  • SAST, dependency scanning, prompt-injection awareness on every AI-touched PR — non-negotiable
  • Measure escape rate to prod, not PR throughput — volume is not a virtue
