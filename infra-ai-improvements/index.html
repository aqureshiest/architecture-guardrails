Example 1

Creating a New RDS Database

Engineer wants a PostgreSQL database for their service. Simple, self-contained. But today it requires a dedicated per-resource action, brittle bash generation, and multiple human gates — starting before the GitHub workflow even fires.

Today — One action per resource type
🔧 actions/create-postgresql-database/action.yml
🔧 actions/create-sqs/action.yml
🔧 actions/create-s3-bucket/action.yml
🔧 actions/legoland-infra-files/action.yml
Each is ~500 lines of bash. Every new resource type requires a new action from scratch.
After — One action for everything
🤖 actions/llm-legoland-writer/action.yml
Inputs: resource-type, params, project-name, environments
Reads legoland context → sends to Claude → writes files.
Works for any resource type, including ones that don't exist yet.
Adding a new resource type is a prompt change, not a new 500-line bash script.

Today — ~2 to 4 hours (elapsed)

Engineer looks up project name, sub-project name, environments. Fills structured form.
Human • ~15 min

Mandatory approval configured on the Port action (requiredApproval: ANY). A designated approver must approve in Port before the GitHub workflow is even triggered. The pipeline is blocked until this happens.
Human • variable (minutes to hours)

create-postgresql-database/action.yml — 500 lines of bash that grep, splice, and string-manipulate postgresql.tf, variables.tf, outputs.tf into legoland. A completely separate action would run for SQS, S3, etc.
Auto • ~2 min
PR Opened Against Legoland
Branch created, PR opened. env0 picks it up and runs tofu plan.
Auto • ~3 min
env0 Plan Review (if env0 approval is enabled)
If the env0 environment is configured to require approval, engineer receives a Slack notification with the plan URL and must navigate to env0 to review and approve. Whether this gate is active depends on env0 environment settings — the workflow warns but cannot enforce it.
Human • ~30–40 min (context switch, when required)
env0 Applies
Tofu apply runs. Workflow polls for terminal status (success/failure). No awareness of mid-run approval states.
Wait • ~10 min

Workflows (Port triggers each separately)

⚙️ orchestrate-service-onboarding.yml

🔧 actions/legoland-infra-files
🔧 actions/create-repo
🔧 actions/env0-project-creator
Two Port forms. Two manual env0 approvals. Each action generates config for one specific concern. Human required to stitch it all together.
After — One agent orchestrates everything
Single unified workflow
⚙️ orchestrate-service.yml
Single LLM action generates all config
🤖 actions/llm-legoland-writer
🤖 actions/llm-plan-reviewer
One intent → agent generates repo scaffold, service infra, K8s wiring, and all resource config in a single pass. Reviews and auto-approves all env0 plans. Merges PRs. Triggers first deploy.
No Port forms. No manual env0 approvals. No PR queue. Human only involved at the start.

Adding a New Self-Service Resource Type

What does it take today to expose a new resource type (e.g. ElastiCache, SNS, RDS cluster) as a self-service action in Port? And what does it take after the AI transformation?

Today — ~1 to 2 weeks of engineering

Every new resource type is a fresh engineering project. No shared infrastructure. No reuse. Each step must be built from scratch.

Step-by-step engineering work
1
Write a new bash composite action
Author actions/create-{resource}/action.yml from scratch. ~500 lines of bash: input validation, file existence checks, grep-and-splice logic for each .tf file the resource needs.
↓
2
Write a new orchestrate workflow
Author orchestrate-{resource}.yml — define all inputs, call the bash action, wire up env0 polling jobs, add Slack notifications for each failure/success state, add DX telemetry.
↓
3
Test against a real legoland project
Manually run the workflow against a test project. Debug bash string-manipulation edge cases. Verify generated .tf files are valid. Fix indentation issues, heredoc escaping, missing variable blocks.
↓
4
Configure Port self-service action
Build a new Port action form: define all input fields, map them to workflow inputs, configure the GitHub webhook trigger. Update Port catalog blueprint if needed.
↓
5
Write documentation + FAQ
Author Confluence page with prerequisites, gotchas, and common failure modes. Engineers need it because the pipeline is opaque.
↓
6
PR review + merge + monitor rollout
Action + workflow PR reviewed by Cloud Platform team. Merged. First few real usages monitored for edge cases missed in testing.
What this produces
✕ One more ~500-line bash action to maintain
✕ One more orchestrate workflow to maintain
✕ One more Port form to maintain
✕ Fragile to legoland convention changes
✕ Engineer must understand bash + HCL + Port + env0
1–2 weeks per resource type ~500 lines of bash per type No reuse between resource types

After AI Transformation — ~1 to 2 hours

The LLM action already exists. The orchestrate workflow already exists. Adding a new resource type is describing what you want — not building new infrastructure to support it.

Step-by-step
1
Describe the resource to the LLM action
Add a brief description of the new resource type to the llm-legoland-writer prompt: what it is, what legoland module backs it, what files it needs. The LLM reads existing legoland examples and infers the rest.
↓
2
Add resource-type to the unified orchestrate workflow
The existing orchestrate-resource.yml accepts a resource-type input. No new workflow needed — just register the new type as a valid value.
↓
3
Test by running it
Trigger the workflow with the new resource type. Review the generated .tf files in the PR. If the output is wrong, refine the description. No bash debugging, no heredoc escaping.
↓
4
Update Port form (optional)
Add the new resource type as an option in the Port form, or accept it as natural language input. Either way, no new Port blueprint needed — the same action handles it.
What this produces
✓ Zero new bash to write or maintain
✓ Zero new workflows to write or maintain
✓ Auto-adapts when legoland conventions change
✓ Engineer only needs to describe intent
✓ Documentation is the PR description, not Confluence
Hours not weeks Zero new bash Fully reuses existing pipeline

Workflow Simplification

How orchestrate-service-onboarding.yml changes

The job graph stays the same — the same 7 jobs, the same sequencing. What collapses is the content of each job. The complexity that disappears is exactly where the LLM takes over.

ℹ️ Same 7 jobs. Same needs: sequencing. ~1050 lines → ~300 lines.
Today — ~1050 lines
config ~370 lines
Bash validates every input: env name formats, replica JSON shapes, image names, env var key=value patterns. Exists because Port form data arrives untrustworthy and needs heavy scrubbing.
adding-vault-role ~170 lines
Creates Vault role, then also runs env0-project-creator and polls for its deployment. Two concerns bundled into one job.
legoland-project-pr ~200 lines
Calls legoland-project-creator (installs cookiecutter + OpenTofu, cherry-picks config commits with retry loops) then chains legoland-infra-files via the post-action-uses hack.
setting-arn-as-output ~85 lines
Fetches env0 Terraform outputs to extract service ARNs. No changes needed.
argo-checking-apps ~45 lines
Polls ArgoCD until healthy. No changes needed.
notify-on-failure / notify-on-success ~180 lines
Summarizes failed jobs, Slacks user, posts to DX. No changes needed.
After — ~300 lines
config ~30 lines
Agent resolves and validates inputs upstream before triggering the workflow. Config job becomes a thin passthrough — echo inputs, set outputs, done.
adding-vault-role ~40 lines
Creates Vault role only. env0 project creation is gone — it's handled as part of the legoland PR that the LLM generates. Concerns are separated cleanly.
legoland-project-pr ~20 lines
One call to llm-legoland-writer. No cookiecutter, no cherry-picking, no retry loops, no post-action-uses chaining. LLM reads context, writes all files, opens PR.
setting-arn-as-output ~85 lines — unchanged
argo-checking-apps ~45 lines — unchanged
notify-on-failure / notify-on-success ~180 lines — unchanged
The three jobs that change account for ~740 lines of complexity today. After the transformation they're ~90 lines combined. The jobs that don't change — polling, notifications, telemetry — are already clean and stay untouched.
