1 Small Build & Test
Standardize linting/formatting scope in CI
Scope ESLint and Prettier checks to changed files only so pre-existing repo-wide issues don't block unrelated PRs. Separate linting failures from hard merge gates until the baseline is clean.
"Checks for Prettier and ESLint are enforced globally… existing issues elsewhere in the repository can still cause the build to fail."Joao Ley · Personal Loans
2 Small Dev Environment
Create / refresh "Run Locally" docs across key repos
Assign a short documentation sprint to each team to audit and update their repo READMEs with local setup steps, required env vars, and known gotchas. Low lift, high onboarding value.
"Running local environment is documented but I found some gaps for setting up completely."Jaime Garcia · Ops Product
"Would be good to prioritize up to date documentation of development workflows for all repositories."Sarah Wesley · New Products
"There is little to no guidance on how to get a service up and running, which forces engineers to rely on trial-and-error or tribal knowledge."Joao Ley · Personal Loans
3 Small Incident Handling
Publish incident response onboarding guide
Publish a central incident response doc with protocols, escalation paths, and a per-service runbook index. Host a lunch-and-learn to walk through it — especially valuable for engineers newer to the org.
"Would love to have onboarding docs, workshop, lunch and learn around how site incidents are handled."Roel Ramirez · New Products
"Not having a clear view with runbooks for incidents for each service/application."Tomas Gutierrez · SRE
4 Small Cross-Team
Establish default code review assignment convention
PRs sit without reviews because no one feels responsible. Add CODEOWNERS files, rotating reviewer assignments, or required reviewer groups so every PR has a named reviewer at open time.
"I think people always expected another person to do the review unless it is assigned."Manuel Zapata · Quality Engineering
"Improve code review turnaround time so I can build momentum and finish my tasks more efficiently."Jeffry Milan · Infrastructure
5 Small Build & Test
Escalate env0 vendor reliability issues
Recurring instability with env0 is blocking Cloud Platform work. Schedule a structured meeting to surface issues, document impact, and define an SLA or mitigation plan.
"Some of our vendors (env0) has lately having issues. It would be great if we could setup a meeting with them to find a way to solve them."Mario Reyes · Cloud Platform
6 Small Production Debugging
Clarify New Relic access management and migration guidance
The Splunk → New Relic migration is causing access gaps with no clear guidance. Publish a transition checklist, assign access owners per service, and communicate the timeline to affected teams.
"There have been instances where access to critical production resources is lost without clear communication or visibility."Joao Ley · Personal Loans
7 Medium Build & Test
Reduce and stabilize flaky E2E tests (Testkube / Cypress)
Flaky e2e tests force 3–5 retries per PR, inflating merge times to over an hour. Quarantine known-flaky tests, add retry budgets with alerting, and investigate staging instability as a root cause.
"e2e tests are very intermittent… intermittent failures mean 3–5 retries are needed, which extends wait times over an hour."Helen Yau · Lending Conversion
"Tests may fail in one run and pass in another… often caused by instability in the staging environment."Dustin Brewer · Growth Products
"Failed tests do not show any logs to debug — only option is to rerun again."Keerthi Hulliyappa · Quality Engineering
8 Medium Build & Test
Optimize CI build times (caching, parallelization, job pruning)
Pipelines routinely take 20–60+ minutes. Audit job graphs across key repos: add dependency caching, parallelize where possible, and make jobs conditional on file changes.
"Some builds are taking anywhere from 20 minutes to over an hour… Ideally, we'd be closer to a 10-minute turnaround."Esteban · Lending Conversion
"The CI process is time consuming. We can eliminate checks that are not necessary."Nihaar Shah · Data Platform
"There are still occasional outages and periods of slowness that impact developer velocity."Lev Solar · Growth Products
9 Medium Build & Test
Harden sandbox environment reliability (Vault / infra)
Vault-related failures in sandbox are causing regular CI pipeline breakage unrelated to code changes. Investigate root cause, add health checks, and implement auto-remediation or clearer failure messaging.
"Our Sandbox uses VAULT and it fails on a regular basis."Gaurav Tamrakar · Ops Product
10 Medium Dev Environment
Standardize local development setup across repositories
Engineers face inconsistent local setup experiences — different Node versions, Docker Compose patterns, and env var management. Build a shared dev container spec, .env.example templates, and a make dev bootstrapping target and roll out to high-traffic repos first.
"Most services require custom changes to run locally… projects should run locally with little or no manual patching."Dustin Brewer · Growth Products
"It's often unclear which Node/runtime versions services expect. Tooling around Chassis / ggo CLIs is inconsistent across repositories."Steven Ciasullo · Growth Products
"Repos have different ways on running locally."Jeu Manzo · Core Lending
11 Medium Production Debugging
Improve observability: logging standards and dashboards
Insufficient logging is making production debugging significantly harder. Define a minimum logging standard (structured logs, request IDs, error context) and create a shared checklist teams use during service reviews. Prioritize services with recent incidents.
"It's common to feel some logs are missing when a production issue happens."Thiago Prochnow · Servicing and Mobile
"Sometimes, applications won't log useful information that helps in debugging."Enrique Cuevas · SRE
"No logs."German Villalpando · User Identity
"We currently lack sufficient observability and well-defined dashboards."Chaitanya Vemulapalli · Growth and New Products
12 Medium Production Debugging
Centralize and align monitoring tooling (New Relic / Splunk)
Monitoring is fragmented across tools with inconsistent alert ownership, creating noise and slowing MTTR. Define a target-state monitoring architecture, set alert ownership rules by team/service, and reduce duplicate or unrouted alerts.
"Monitoring could be more effective if more centralized and fully aligned across New Relic and Splunk. Alerts not scoped to the right team create noise."Colin Enos · Core Lending
"Having a better picture of dependencies and what-talks-to-what would go a long way toward reducing MTTR."Andrew Templin · SRE
13 Medium Deep Work
Reduce meeting load and protect focus time
Multiple teams report excessive meetings cutting into deep work time. Audit recurring meetings for ROI, expand no-meeting blocks beyond Tuesday, and establish async-first norms for ceremonies that can be replaced with written updates.
"I only really have one day a week (no-meeting Tuesday) that allows for deep focused work time."Mark Taylor · Infrastructure
"I'm increasingly in recurring meetings for agile ceremonies and syncs that aren't bringing net value."Sean Galloway · Core Lending
"Still balancing a large amount of meetings and last minute syncs between focus work."Lauren Childs · Analytics
14 Medium Decision-Making
Define lightweight decision-making tiers
Implementation-level decisions are being escalated into full design review cycles. Define a taxonomy: what an individual engineer can decide, what needs team alignment, and what requires architectural review. Document and socialize it.
"Many implementation-level details are being treated as design decisions… introduces unnecessary discussions, approvals, and alignment cycles."Chaitanya Vemulapalli · Growth and New Products
"It could help to quickly assess the blast radius of a decision before beginning discussions."Sean Galloway · Core Lending
15 Medium Cross-Team
Improve cross-team PR review visibility and SLAs for shared repos
Review delays on cross-team PRs slow teams with domain context but no authority to merge. Define review SLAs per repo type, expand CODEOWNERS where teams have demonstrated context, and create a shared PR review queue dashboard.
"Small Looker changes used to move quickly… now some PRs can take up to ~2 hours. The analytics team has strong context on these models."Haley Mosdell · Analytics
"All teams seem to have very full workloads right now which can make it difficult to get responses/input."Kyle Roark · Data Platform
"Given the amount of code we can generate with AI today, the bottleneck has become reviewing the code."Pedro ortega.martinez · Growth Products
16 Large Incident Handling
Build a service dependency map and request-flow catalog
SREs and engineers consistently lack visibility into how services communicate, making incident diagnosis and change coordination significantly harder. Build and maintain a service dependency graph (automated discovery + visual catalog in Backstage or Confluence) covering services, dependencies, and owning teams.
"Having a better picture of dependencies and what-talks-to-what would go a long way toward reducing MTTR."Andrew Templin · SRE
"As an SRE, it's hard to understand the flow of the requests between applications."Enrique Cuevas · SRE
"Multiple teams making changes in the same repositories can make it difficult to have full visibility into related changes."Colin Enos · Core Lending
17 Large Build & Test
Establish a test ownership and quality maintenance framework
Performance tests and test suites are becoming outdated as the product evolves, reducing CI signal quality. Define a test ownership model per team, introduce flakiness rate and coverage drift metrics into team dashboards, and make test health a sprint ritual.
"The automated test suite, particularly performance tests, is not consistently maintained… reduces confidence in CI signals and slows delivery."Chaitanya Vemulapalli · Growth and New Products
"We need to define better our tests."Alejandro Estrada · User Identity
"I want to see our code has full test coverage and meaningful checks."Thien Phan · Security
18 Large Incident Handling
Improve incident response: automation, runbooks, and ownership routing
Incident response is hampered by unclear ownership and missing runbooks. Build: (1) per-service runbooks linked from alerting tools, (2) automated routing that pulls in recent deployers for deployment-related incidents, (3) a post-incident review process that feeds improvements back to service teams.
"Response workflows could be improved by automatically involving recent deployment contributors and reducing operational noise through better routing."Colin Enos · Core Lending
"Not having a clear view with runbooks for incidents for each service/application."Tomas Gutierrez · SRE
"The incidents that occurred in early March highlighted some areas where we can improve."Kyle Roark · Data Platform
19 Large Ease of Release
Reduce multi-team deployment friction in shared repositories
Multiple teams sharing a single repo and deploy pipeline leads to canceled or delayed deploys. Investigate feature flag infrastructure and independent deploy strategies (per-team deploy trains, trunk-based development with flags) to decouple team release cadences.
"Multiple teams working in single repositories has led to friction in our deploy schedules. Juggling changes has often led to later than desired and even canceled deploys."Kyle Woolley · Personal Loans
"It's been challenging following the CLP release schedule. These restrictions create a sense of rush."Helen Yau · Lending Conversion
"Reoccurring code freezes make it difficult to be confident about changes."Arjun Gopisetty · Infrastructure
20 Large Dev Environment
Invest in a developer portal / internal docs platform (Backstage or equivalent)
Multiple pain points — inconsistent local setup, unclear service ownership, missing runbooks, undocumented dependencies, poor cross-team visibility — share a common root: no central source of truth. Evaluate and adopt an internal developer portal to catalog services, owners, docs, runbooks, and setup guides in one discoverable place.
"There is little to no guidance on how to get a service up and running, which forces engineers to rely on trial-and-error or tribal knowledge."Joao Ley · Personal Loans
"Still a pain point to not have visibility in other teams' projects."Enrique Gonzalez · Quality Engineering
"Code repos felt like a 'bend don't break' mentality… areas of improvement include test suites, README documentation, and general upkeep."Roel Ramirez · New Products
21 Large Code Maintainability
Tackle chassis abstraction complexity and code debt
Engineers struggle to reason about code due to layered abstractions, undocumented chassis opinions, and accumulated tech debt. Launch a structured tech debt reduction initiative: identify highest-friction abstractions, document intended behavior, simplify where possible, and add guardrails to prevent re-accumulation.
"Some of the abstractions we've built on top of the chassis make comprehension harder to reason about."Chris Ciervo · Personal Loans
"There is a lot of complexity that takes a lot of thinking to get through to be able to make a required refactor."Adan Paino · Cloud Platform
"Recent initiatives have involved shoring up long standing tech debt that involves a chained dependency tree of needed updates."Matt Kirchstein · Web Platform
22 Large Ease of Release
Enable safe production testing (test accounts / safe test data)
Engineers lack a safe way to validate certain flows in production, forcing teams to either block deploys or "fly blind." Design and implement a production test account framework with safe test data (test SSNs, synthetic accounts) to allow controlled validation without impacting real users.
"Developer productivity would increase if there were a way to create test accounts on Production. There are a few instances where we have to 'fly blind'."Roel Ramirez · New Products
"Improve ability to test production flows using safe test data (e.g., test SSNs)."Lev Solar · Growth Products
